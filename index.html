Architecture deep-dives

Three systems built at Warner Bros. Discovery (Food Network, CNN, Eurosport, and others) — internal service names are not disclosed. Plus one open-source personal project with a full diagram.

Privacy / PII Request Orchestrator

Compliance · GDPR Article 15 + 17 · Async fan-out across 6+ services

What it is. A user privacy request (right of access or right to erasure) enters via OneTrust and is forwarded by a third-party broker to the Compliance Request Orchestrator. The orchestrator looks up which downstream services hold PII for that user via a service registry, then fans out across a Kafka request stream. Each downstream consumes its slice, processes (delete or export), and publishes a per-service result. The orchestrator aggregates results and calls the broker's fulfillment API once the aggregate state flips to complete.

Why this shape. The subject-request-ID drives idempotency end-to-end, so re-deliveries at any hop are safe. A service registry — rather than hardcoded routing — lets new PII-holding services join compliance by registering, with no orchestrator change. Two separate Kafka streams (request out, result in) give each downstream a clean inbound contract and the orchestrator a clean aggregation contract. State is split into per-service status and an aggregate; the aggregate flips to complete only when every individual settles, gating the fulfillment callback.

Trade-offs. Aggregating across N independent consumers makes overall latency a function of the slowest downstream — acceptable for a regulator-deadline-bounded workflow (days, not seconds), and worth it for the auditability and per-service ownership clarity.

Multi-Brand Async Moderation Platform

Trust & Safety · Multi-tenant · In-house text + image moderation

What it is. User-generated content — comments, nicknames, image uploads — from multiple brand surfaces flows through an internal API gateway, is identity-checked via CIAM, then published to a Kafka request stream. An inbound Lambda dispatches to two in-house classifiers: a text moderation processor for comments and nicknames, and a media processor for images. Both publish verdicts to a separate Kafka response stream; an outbound Lambda picks up each verdict and notifies the originating brand surface. Image bytes are stored in S3; record metadata and S3 keys live in DynamoDB.

Why this shape. Two streams (request out, response in) localise replay — a failure in moderation never re-drives the entire pipeline. Both classifiers are owned in-house: text moderation was migrated off a third-party vendor, image moderation absorbed from a sister team, eliminating vendor cost and returning policy ownership to the product team. Per-brand rules (banned terms, severity thresholds) are hot-reloadable without redeployment.

Roadmap. Soft-publish — content visible only to its author until a non-violation verdict arrives — is a planned state-machine extension; today verdicts are applied retroactively.

Backfill Search-Indexing Pipeline

Data engineering · Offline ETL · Zero impact on production DynamoDB

What it is. A one-time backfill that lifts a large content corpus out of production DynamoDB into OpenSearch without touching production read capacity. DynamoDB's native S3 export feeds three domain-partitioned source buckets (Items, Collections, Mappings); an AWS Glue job validates, filters, transforms, and partitions the data into Parquet (Snappy) on S3. An S3 event-per-object triggers a Node.js Lambda that parses Parquet, validates rows, batches ~1000 records at a time, and POSTs them to a search-indexing API in front of OpenSearch. Exhausted retries route to an SQS FIFO dead-letter queue for manual replay.

Why this shape. A naïve table scan would burn DynamoDB read capacity and degrade product latency. The native S3 export decouples extraction from production load entirely — the rest of the pipeline is a normal S3-driven ETL. Parquet+Snappy is fast to parse in Lambda and partition-pruning friendly. The ~1000-record batch is sized against both the OpenSearch bulk-index sweet spot and the Lambda execution-time budget, avoiding timeouts on outliers.

Trade-offs. Manual DLQ replay is acceptable for a one-time backfill — automating it wasn't worth the engineering cost for a single run. Dedup on the index side isn't yet implemented; an op_type=upsert with deterministic doc IDs is the next step if this evolves into an ongoing pipeline.

Full Budget App — Open Source

Serverless · Multi-tenant · Event-driven categorisation · AWS CDK · github.com/debrajpaul/full-budget-app

What it is. A multi-tenant serverless budgeting platform built as a pnpm monorepo on AWS. Users upload bank statements (HDFC, SBI, Axis — PDF or CSV) via a GraphQL mutation; the statement lands in S3, an S3 event queues a message in SQS, and the txn-loaders Lambda parses and normalises the transactions into DynamoDB. A DynamoDB stream on the Transactions table immediately triggers the tag-loaders Lambda, which applies a deterministic NLP rule engine for known merchants and categories, falling back to Amazon Bedrock (Mistral) only for low-confidence matches. All five DynamoDB tables use tenantId as the partition key, giving hard data-boundary isolation between users.

Why this shape. S3 → SQS decouples ingestion from parsing — spiky upload volumes don't block the GraphQL API. DynamoDB Streams replace polling entirely: as soon as a transaction is written, the categorisation worker fires. The hybrid NLP+AI strategy keeps Bedrock calls (and their cost) to a minimum; most common merchants are classified by rules in under a millisecond. AWS CDK provisions all resources — tables, queues, buckets, alarms, X-Ray groups — as reproducible stacks with explicit dependency ordering.

Trade-offs & next steps. Bedrock IAM is currently unrestricted; restricting it to specific model ARNs is the obvious next hardening step. DynamoDB removal policies are set to DESTROY for dev convenience — switching to RETAIN is required before any production use. Test coverage is scaffolded but not yet wired into CI; adding integration tests against LocalStack is the next quality gate.
